# 📅 2026-03-24 (Phase 1 — 1주차 개발 상세 워크스루)

## 📌 개요
오늘 1주차 개발 세션에서는 **LangGraph 및 MCP 기반 추론-실행 에이전트(Agent)의 코어 뼈대**를 구축하고, **VLM 연동**, **Office 자동화(Word CRUD)**, 그리고 **하드웨어 Kill Switch**를 완벽하게 연동했습니다. 

총 17개의 핵심 코드 파일과 각 모듈을 이어주는 10개의 `__init__.py` 파일을 생성했으며, 각 파일을 어떻게 생성했는지, 코어 로직은 무엇인지, **왜 이렇게 설계해야만 했는지**를 아래에 상세하게 기재합니다.

---

## 🛠️ 컴포넌트별 생성 파일 상세 내역

### 1. `config/` (전역 설정 컴포넌트)
> **도입 이유**: 하드코딩된 값들이 여러 모듈에 흩어지는 것을 방지하고, 타입 안정성(Type Safety)을 보장하기 위해 중앙 집중식 설정/타입 계층을 두었습니다.

- **`config/settings.py`**
  - **어떻게?**: `pydantic-settings`의 `BaseSettings`를 상속받아 싱글톤 패턴(`@lru_cache`)으로 구현했습니다.
  - **왜?**: `.env` 파일과 시스템 환경 변수를 자동으로 병합하고 검증해 주며, 에이전트 생성 과정에서 설정값이 꼬이지 않도록 단일 진실 공급원(SSOT) 역할을 수행하게 끔 하였습니다.
- **`config/types_dto.py`**
  - **어떻게?**: `pydantic` v2 기반으로 `TaskRequest`, `ScreenState`, `ToolInfo` 등의 교환 데이터(DTO) 템플릿을 정의했습니다.
  - **왜?**: 각 계층(Sensor ↔ Brain ↔ Embodiment) 간의 통신 시 주고받는 데이터의 포맷을 명확히 규정하여 딕셔너리(`dict`) 사용 시 발생하는 Key 에러와 타이핑 불명확성을 원천 차단하기 위함입니다.
- **`config/constants.py`**
  - **어떻게?**: Python 표준 라이브러리인 `enum.Enum`을 상속받아 `AgentStatus`, `ToolCategory`, `ErrorCode` 등의 상수로 정의했습니다.
  - **왜?**: 문자열 매직 넘버 사용을 지양해 오타를 방지하고 자동완성 기능의 도움을 받아 안정성을 증대시키기 위함입니다.

### 2. `utils/` (공통 유틸리티 컴포넌트)

- **`utils/logger.py`**
  - **어떻게?**: `loguru` 라이브러리를 사용해 기본 `logging`을 대체하며, 파일(`data/logs/`)과 콘솔(`sys.stderr`) 양방향으로 동시 출력되도록 `setup_logger()` 함수를 구현했습니다.
  - **왜?**: `print()` 문으로 디버깅하는 악습을 차단. 디버깅, 사용자 알림, 추후 감사 로깅 시스템까지 하나의 통일된 메커니즘으로 일원화하기 위해 도입했습니다.

### 3. `embodiment/` (OS 제어 및 안전 시스템)
> **도입 이유**: 에이전트가 소프트웨어나 하드웨어를 '직접' 조작하는 물리적인 수족 역할을 합니다.

- **`embodiment/kill_switch.py`**
  - **어떻게?**: `pynput` 마우스 리스너 스레드를 백그라운드에 띄워 1.5초 내에 마우스 **우클릭이 10회** 감지되면 `os._exit(1)`로 즉시 프로젝트를 강제 종료하도록 구현했습니다.
  - **왜?**: AI가 Windows API를 제어하거나 마우스를 제어할 때 무한 루프에 빠지거나 엉뚱한 화면을 조작하는 경우, 운영체제 단위의 심각한 손실을 초래할 수 있으므로 0순위로 **가장 빠르고 물리적인 비상 정지 수단**을 만들기 위함입니다.
- **`embodiment/office_toolkit.py`**
  - **어떻게?**: `pywin32` 패키지의 `win32com.client`를 사용해 Microsoft Word COM 객체(`Word.Application`)에 실시간 직접 연결되도록 개발했습니다. 특히 문서가 조작되는 과정을 볼 수 있게 `Visible=True`를 할당했습니다.
  - **왜?**: `python-docx` 같이 파일 자체를 수정하는 정적 방식보다, 실제로 메모리에 열려 있는 오피스 프로그램 창을 실시간으로 가져와 에이전트가 사용자 대신 타이핑하듯 역동적인 조작을 수행하기 위함입니다.

### 4. `strategy/` (도구 통합 관리)
> **도입 이유**: 에이전트가 쓸 수 있는 무기(Tool)들을 종류에 상관없이 일관된 형태로 LangGraph에게 제공해야 합니다.

- **`strategy/local_tools.py`**
  - **어떻게?**: LangChain의 `@tool` 데코레이터를 이용해 Word 문서 CRUD(생성, 읽기, 변경, 삭제) 기능 7개를 감싸 도구 명세(docstring)와 인자 스키마를 정의했습니다. (특히, 경로 생성 시 `data/results`를 무조건 참조하도록 시스템 프롬프팅 적용)
  - **왜?**: LLM은 도구의 이름과 docstring만을 보고 실행 여부를 판단하므로(Tool Calling), 도구 사용 목적과 제약사항을 명확히 설명해두기 위함입니다.
- **`strategy/tool_registry.py`**
  - **어떻게?**: `register_tools`와 `get_tools` 인터페이스를 가진 메모리 레지스트리를 클래스로 설계했습니다.
  - **왜?**: 향후 수십 개의 로컬 도구와 원격 MCP 도구들이 섞일 텐데, 도구의 위험 등급(Risk Level)에 따라 특정 도구는 사용을 차단하는 식의 **안전 통제 게이트웨이**를 구현하기 위해서입니다.
- **`strategy/mcp_client.py`**
  - **어떻게?**: `langchain-mcp-adapters`를 이용해, 향후 외부 Node.js 기반 파일 조작이나 GitHub API 조작 툴 등을 동적으로 불러오도록 비동기 컨텍스트(`AsyncContextManager`) 기반 연결부를 준비했습니다.

### 5. `brain/` (인지 및 추론 - LangGraph 코어)
> **도입 이유**: 단발성 질의응답이 아닌, 계속해서 생각-행동을 반복하는 **순환 인지 루프**를 관장합니다.

- **`brain/graph_state.py`**
  - **어떻게?**: LangGraph의 꽃인 그래프 상태 객체를 파이썬의 `TypedDict` 형태로 정의했습니다 (`messages`, `iteration_count`, `action_history` 등 보관).
  - **왜?**: 여러 상태 노드들을 통과하는 내내 모든 과거 기록과 대화 컨텍스트가 유실되지 않도록 보존하기 위해서입니다.
- **`brain/prompts.py`**
  - **어떻게?**: 에이전트의 성격과 역할을 규정하는 `SYSTEM_PROMPT`와, 파일 저장 시 `data/results` 절대경로를 사용해야 한다는 핵심 행동 수칙을 담은 문자열 포매팅으로 구성했습니다.
  - **왜?**: 프롬프트만 전문으로 관리하는 곳을 따로 두어, 튜닝 시나 유지보수 시 코드(로직)와 텍스트(성향)를 분리하기 위함입니다.
- **`brain/nodes/reasoning_node.py`**
  - **어떻게?**: 그래프 상태 변수를 받아와서 프롬프트와 결합한 뒤, `qwen3-vl:8b` 로컬 Ollama 모델 인스턴스에 던져 추론 결과(도구 실행을 요청하는 `AIMessage`)를 받아오는 노드를 구성했습니다.
  - **왜?**: LLM의 사고 과정만을 전담하는 모듈이 어야 하며, 여기서 나온 `tool_calls` 여부에 따라 LangGraph가 다음 노드를 분기할 수 있게 하기 위함입니다.
- **`brain/nodes/tool_node.py`**
  - **어떻게?**: Reasoning 노드에서 지시한 도구 이름(예: `create_word_doc`)을 파싱하여 실제 `ToolRegistry`에서 객체를 가져와 실행하고 그 결과값을 `ToolMessage`로 포장하는 노드를 구성했습니다.
  - **왜?**: 외부 애플리케이션에 영향을 미치는 부수 효과(Side-Effect)는 이 노드에서만 집중적으로 발생토록 하여, 권한이나 오류 관리를 일원화하기 위함입니다.
- **`brain/graph_builder.py`**
  - **어떻게?**: `StateGraph`에 위 두 노드(`reasoning`, `tools`)를 추가하고, LLM이 도구 사용을 요구하면 `tools` 노드로 가고, 아니면 종료(`END`)하는 조건부 간선(Conditional Edge)을 선언했습니다.
  - **왜?**: '생각 → 수행 → 성공여부 재평가 → 다시 생각'이라는 사람과 비슷한 업무 수행 반복 프로세스를 LangGraph 엔진을 통해 구현하기 위함입니다.

### 6. 시스템 진입점 및 환경
- **`main.py`**
  - **어떻게?**: `argparse`를 이용해 CLI 명령어를 받아, `KillSwitch` 스레드를 백그라운드에 가동시킨 채 앞서 만든 LangGraph 에이전트에 입력을 주입해 E2E(End-to-End) 실행을 관장하도록 작성했습니다.
  - **왜?**: 사용자가 `python main.py` 하나만으로 환경 변수를 로드하고 패키지를 조립해 즉시 에이전트와 대화형 또는 단일 명령 방식으로 쉽게 상호작용하도록 추상화하기 위함입니다.
- **`.env`**
  - **어떻게?**: `dotenv` 패키지를 염두에 두고 비밀번호성 정보나 로컬 설정(`qwen3-vl:8b`, `data/results` 등)을 격리시킨 환경변수 템플릿입니다.
  - **왜?**: 향후 다른 PC나 Git 클론 시 사용자만의 IP 주소, API 키 설정 등이 코드에 하드코딩되어 유출되거나 충돌하는 문제를 막기 위함입니다.

## ✅ 요약 결론
1주차에 생성된 이 **17개의 핵심 파일**은 철저하게 **단일 책임 원칙(SRP)**과 **레이어드 아키텍처** 패턴을 지키며 설계되었습니다. 
각 모듈이 분리되어 있으면서도 서로의 책임을 명확히 오가(Sensors → States → Brains → Embodiments)도록 했으며, 이를 통해 향후 Phase 2에서 PySide6 UI나 TTS 연동 시 **기존 에이전트 핵심 로직을 전혀 뜯어고치지 않고도 확장이 가능하도록(OCP)** 유연한 기반을 오늘 최종 완성했습니다.
